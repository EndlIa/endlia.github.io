## 题面

我们已经学习了许多可以用于程序性能优化的手段。现在，你可以采取任何策略来优化 `arraymul` 函数，包括但不限于循环不变式外提、循环展开、SIMD、多线程等等。

`arraymul()` 的 baseline 实现如下：

```c
#include "student.h"

typedef struct{
    size_t len;
    int *data;
} vec;

void arraymul(vec * v1, vec * v2, vec *v3){
    for (size_t i = 0; i < v1->len; i++) {
        v3->data[i] = v1->data[i] * v2->data[i];
    }
}
```

其中，`student.h` 的内容为：

```c
#include <stddef.h>
#include <stdlib.h>
#include <time.h>
#include <sys/time.h>
#include <stdio.h>
#include <immintrin.h>
#include <x86intrin.h>
```

注意，不需要做错误处理。评测保证传入 `arraymul` 的 `v1`、`v2`、`v3` 的 `len` 都是一致的，且保证乘法不会溢出。

评测时使用的编译选项为：

```text
-w -mavx2 -O0 -pthread
```

需要合理选择优化策略，且不要尝试修改编译选项。


## baseline 慢在哪里

baseline 每轮循环都要做这些事情：

1. 读取 `v1->len`
2. 读取 `v1->data`
3. 读取 `v2->data`
4. 读取 `v3->data`
5. 计算数组下标
6. 做一次乘法和一次写回

在 `-O0` 下，编译器通常不会把 `v1->len`、`v1->data` 这些字段缓存到寄存器里，也不会主动展开循环或向量化。因此这个循环虽然逻辑简单，但每次迭代都有不少额外开销。

第一步优化应该是把循环不变量取出来：

```c
size_t len = v1->len;
int *a = v1->data;
int *b = v2->data;
int *c = v3->data;
```

## 为什么 SIMD 是核心策略

题目允许使用 `student.h`，里面已经包含了：

```c
#include <immintrin.h>
#include <x86intrin.h>
```

同时编译选项有：

```text
-mavx2
```

这意味着我们可以直接使用 AVX2 intrinsic。

`int` 是 32 位，AVX2 的 `__m256i` 是 256 位，所以一次可以处理 8 个 `int`：

```c
__m256i va = _mm256_loadu_si256((__m256i *)(a + i));
__m256i vb = _mm256_loadu_si256((__m256i *)(b + i));
__m256i vc = _mm256_mullo_epi32(va, vb);
_mm256_storeu_si256((__m256i *)(c + i), vc);
```

## 使用 loadu/storeu

上面的代码使用的是 `_mm256_loadu_si256` 和 `_mm256_storeu_si256`。名字里的 `u` 是 unaligned，意思是不要求内存地址对齐。

所谓对齐，是指一个内存地址满足某个对齐粒度的整数倍要求。在这里，AVX2 的 `__m256i` 是 256 位，也就是 32 字节；如果使用要求对齐的加载/存储指令，那么被读取或写入的起始地址需要是 32 字节的整数倍。

举个例子，假设数组首地址是 `0x1000`，它能被 32 整除，那么从这里读取一个 `__m256i` 就是 32 字节对齐读取：

```text
0x1000, 0x1020, 0x1040, ...
```

但如果数组首地址是 `0x1004`，那么从这里读取 32 字节就不是 32 字节对齐的：

```text
0x1004, 0x1024, 0x1044, ...
```

为什么要关心对齐？因为 CPU 和缓存系统搬运数据时通常按固定大小的块工作。地址对齐时，一次向量加载更可能落在一个自然的内存块里；地址不对齐时，一次 32 字节读取可能跨过两个内存块，硬件处理起来更麻烦。在一些旧指令或旧平台上，使用要求对齐的加载指令去读未对齐地址，甚至可能直接出错。

AVX2 里有两类常见加载/存储接口：

- `_mm256_load_si256` / `_mm256_store_si256`：要求地址 32 字节对齐。
- `_mm256_loadu_si256` / `_mm256_storeu_si256`：不要求地址 32 字节对齐。

这道题没有保证 `v1->data`、`v2->data`、`v3->data` 的地址一定是 32 字节对齐的。如果强行使用对齐版本，就必须额外处理数组开头的若干元素，让后面的指针走到 32 字节边界；而且三个数组的起始地址还不一定能同时对齐。

所以这里选择 `loadu/storeu` 更稳：它不依赖额外的地址假设，代码更简单，正确性风险更低。现代 x86 处理未对齐访问的能力已经比较好，只要不是频繁跨缓存行或页边界，通常不会比对齐版本差太多。

## 是否需要多线程

题目编译选项里有 `-pthread`，理论上可以多线程。但是这道题不一定适合一上来就用线程。

原因是：

- 如果数组较小，创建线程的开销可能比计算本身还大。
- 评测可能会调用很多次函数，频繁创建线程会非常亏。
- 题目没有说明运行环境有几个核心，也没有说明数据规模。
- `arraymul` 是纯内存流式访问，也就是说本身的计算量不大，很多时候瓶颈会很快变成内存带宽。

如果已知评测数组极大，再考虑固定线程池或分块多线程。但在这个题目给出的接口中，每次调用 `arraymul` 都临时建线程，未必划算。

## 实现

```c

void arraymul(vec * v1, vec * v2, vec *v3){
    size_t len = v1->len;
    int *a = v1->data;
    int *b = v2->data;
    int *c = v3->data;

    size_t i = 0;
    size_t limit = len & ~(size_t)7;

    for (; i < limit; i += 8) {
        __m256i va = _mm256_loadu_si256((const __m256i *)(a + i));
        __m256i vb = _mm256_loadu_si256((const __m256i *)(b + i));
        __m256i vc = _mm256_mullo_epi32(va, vb);
        _mm256_storeu_si256((__m256i *)(c + i), vc);
    }

    for (; i < len; i++) {
        c[i] = a[i] * b[i];
    }
}
```

## 还能不能继续优化

可以考虑循环展开，例如一次处理 16 或 32 个 `int`：

```c
for (; i + 15 < len; i += 16) {
    __m256i a0 = _mm256_loadu_si256((const __m256i *)(a + i));
    __m256i b0 = _mm256_loadu_si256((const __m256i *)(b + i));
    __m256i c0 = _mm256_mullo_epi32(a0, b0);
    _mm256_storeu_si256((__m256i *)(c + i), c0);

    __m256i a1 = _mm256_loadu_si256((const __m256i *)(a + i + 8));
    __m256i b1 = _mm256_loadu_si256((const __m256i *)(b + i + 8));
    __m256i c1 = _mm256_mullo_epi32(a1, b1);
    _mm256_storeu_si256((__m256i *)(c + i + 8), c1);
}
```

循环展开可以减少分支判断和循环变量更新的次数，也能给 CPU 更多独立指令。但展开太多会让代码变长，在 `-O0` 下也可能带来寄存器分配和指令缓存方面的问题。

寄存器压力是这里需要控制的一个点。未展开时，一轮循环大致只需要 `va`、`vb`、`vc` 这几个向量临时值；展开到 4 路后，一轮循环里会出现 `a0`、`b0`、`r0` 到 `a3`、`b3`、`r3` 这样的多组临时值。CPU 的向量寄存器数量有限，如果同一时间需要保留的临时值太多，编译器可能不得不把一部分值临时放到栈上，之后再从内存读回来。这种寄存器溢出会引入额外的内存读写，抵消循环展开省下来的分支开销。

当然，这段代码每组 load、mul、store 之间依赖很短，编译器理论上可以复用寄存器，不一定真的让所有临时变量同时活着。但评测使用 `-O0`，编译器不会做太积极的寄存器分配和调度，所以展开过度的风险比 `-O2`、`-O3` 下更高。

这里说的寄存器数量，主要指编译器能直接使用的架构寄存器。以 x86-64 + AVX2 为例，程序能直接命名的 256 位向量寄存器通常是 `ymm0` 到 `ymm15`，也就是 16 个。现代 CPU 内部为了乱序执行和寄存器重命名，确实还会有更多物理寄存器，但这些物理寄存器不是 C 程序或汇编代码能直接指定的资源。编译器生成的代码仍然只能使用架构暴露出来的寄存器名；如果这些架构寄存器不够容纳当前活跃的临时值，就仍然可能发生 spill。

这题比较合理的取舍是展开到 16 个或 32 个 `int`，不要无限展开。
